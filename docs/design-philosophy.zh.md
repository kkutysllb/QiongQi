# Qiongqi 设计哲学

> 穷奇非凶，乃破局之锐。不唯上，不唯书，只唯实。
> 让 Agent 如凶兽般凶猛进化，在混沌中去中心化地达成共识。

本文档阐述 Qiongqi 的设计哲学。它由三段神话原型出发，分别映射到引擎
的三个核心主张；继而给出主张与当前架构的真实吻合程度；最后留下一份
"叙事→实现"差距待办，作为未来重构的可选 backlog。

## 一、神话原型

### 1. 核心哲学：打破"中央集权"的叛逆者（对应去中心化）

> 传统神话中，穷奇最大的"恶"是"毁信废忠"（出自《左传》）——它不忠于
> 单一的天庭秩序，不信奉既定的善恶规则。

> 映射到引擎：这正是我们设计的灵魂。穷奇引擎拒绝"主控 Agent"的寡头
> 模式。传统框架往往依赖一个中央调度器，而穷奇认为，真正的智能不应
> 被中心节点束缚。每个 Agent 都是"少昊氏的不才子"，不盲从权威，只
> 忠于任务本身，从而在架构上彻底实现了去中心化的"信"与"忠"。

### 2. 个体能力：独当一面的"虎翼"（对应端到端独立性）

> 《山海经·海内北经》描述穷奇"状如虎，有翼"，能独自捕猎，且"食人
> 从头始"——这意味它拥有完整的捕猎闭环逻辑，不依赖他人投喂。

> 映射到引擎：我们将每个 Agent 设计为具备完整的感知-决策-执行闭环
> （端到端）。它像插上翅膀的猛虎，哪怕脱离协作网络，单兵空降到任意
> 复杂业务场景中，都能凭借内置的推理能力独立啃下硬骨头，具备极强
> 的原子化生存与作战能力。

### 3. 群体智能：懂"人话"的自协商组织（对应组合协作与解耦）

> 《神异经》记载穷奇"知人言语"，且具有"惩善扬恶"的叛逆逻辑——我们
> 重新解构这一点：它不盲从表面的"善"（预设规则），而是通过听懂彼此
> 的"语言"，动态评估当前局势的"利害得失"。

> 映射到引擎：当多个穷奇 Agent 组合时，它们会像人类团队一样自协商。
> 谁擅长数据分析（财务部），谁擅长调用 API（执行部），引擎本身不参与
> 业务决策，只提供"议事规则"。正因为引擎层与技能体系完全解耦，穷奇
> 可以像给不同"公司"注入灵魂一样，今天挂载金融技能包就是风控团队，
> 明天挂载 AIGC 技能包就是创意工作室——引擎是骨架（穷奇本身），
> 技能是血肉（外部组织），两者通过标准化接口交融，既保持了神话凶
> 兽的灵活凶猛，又具备了现代企业的模块化合规性。

## 二、主张 ↔ 当前架构映射（自评）

| 主张 | 当前实现 | 吻合度 |
| --- | --- | --- |
| ① 去中心化：拒绝中央调度器，每个 Agent 忠于任务本身 | `TurnOrchestrator`（`src/loop/turn-orchestrator.ts`）+ `runtime-factory.ts` 仍是显式的中央编排；子代理通过 `DelegationRuntime` 受控派生 | ⚠️ 部分吻合。子代理有自决权，根回合仍由中心协调 |
| ② 端到端独立：单兵闭环作战 | 每个 thread 拥有独立 `messages.jsonl` + `events.jsonl` + `session.json`；`buildDefaultLocalTools` 提供感知-决策-执行完整工具集；`LocalToolHost` 不依赖远端调度 | ✅ 高度吻合 |
| ③ 引擎与技能解耦，模块化挂载 | `CapabilityRegistry`、`SkillRuntime`、`SkillPluginHost`、`MCP tool provider`、`Web tool provider`、`Memory tool provider`、`Delegation tool provider` 全部以 provider 形式热插拔；通过 `skill-mcp-bridge` 还能把 skill 自身声明的 MCP 服务合并进注册表 | ✅ 完全吻合 |

## 三、叙事 → 实现差距待办

本节列出"主张在叙事上成立、但代码尚未完全支撑"的项目，作为后续
重构的可选 backlog。本文档不强制任何具体实现路径：

- **去中心化的根回合编排**：`TurnOrchestrator` 当前仍是单进程内的
  显式协调器。要让"穷奇"在结构层面成立，可考虑把回合推进拆为多个
  peer-style 协作者（`PromptBuilder / ModelStepRunner / ContinuationPolicy
  / ToolCallCoordinator` 之间通过事件流而非显式调用串联），并允许子代
  理以独立身份而非 `DelegationRuntime` 的派生结果出现。
- **跨进程的 Agent 寻址**：当前所有 Agent 都共享同一 Node 进程与同一
  `dataDir`。要做到"任意 Agent 都能被其他 Agent 寻址"，需要定义一套
  跨进程的 capability 寻址协议（很可能以现有 MCP + capability registry
  为底座，但需要明确的 peer identity 与可达性声明）。
- **议事规则的边界**：当前 `ContinuationPolicy` / `tool-call-coordinator`
  已经把"何时停、何时批、何时打断"做成显式策略，但引擎仍会兜底
  处理某些边界（rate limit、storm breaker）。要让"引擎不参与业务决策
  只提供议事规则"成立，需要把所有兜底逻辑显式 policy 化。

> 上述 backlog 仅作叙事登记，**不在本次改动中实施**。任何对运行时
> 行为的修改都需要独立的设计讨论、spec、plan 与回归测试。
